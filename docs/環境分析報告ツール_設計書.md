## 2. データ構造

### 2.1 元データCSV（analysis_data.csv）

分析結果の全データ。業務ツール起動時に最新版を読み込む。

| カラム（想定） | 説明 |
|---|---|
| job_number | JOB番号（報告書の識別キー1） |
| protocol_name | 依頼時プロトコル名（報告書の識別キー2） |
| sample_id | サンプルID |
| sample_name | サンプル名 |
| ... | 分析結果の各項目 |

### 2.2 設定ファイル（config.json）

業務ツールがローカルに保持する設定ファイル。exe と同じディレクトリに配置。

```json
{
  "version": "1.0",

  "sharepoint_paths": {
    "source_dir": "C:/Users/username/会社名/報告ツール/source",
    "reports_dir": "C:/Users/username/会社名/報告ツール/reports"
  },

  "report_definitions": [
    {
      "report_id": "RPT-001",
      "report_name": "定期環境分析報告",
      "search_filters": {
        "protocol_name": ["水質分析プロトコルA", "水質分析プロトコルB"]
      },
      "description": "月次の水質分析報告"
    },
    {
      "report_id": "RPT-002",
      "report_name": "排水管理報告",
      "search_filters": {
        "protocol_name": ["排水分析プロトコル"]
      },
      "description": "排水基準値の確認報告"
    }
  ],

  "departments": [
    {
      "dept_id": "DEPT-A",
      "dept_name": "製造1課",
      "folder_name": "製造1課",
      "allowed_samples": {
        "RPT-001": ["SAMPLE-001", "SAMPLE-002", "SAMPLE-003"],
        "RPT-002": ["SAMPLE-010", "SAMPLE-011"]
      }
    },
    {
      "dept_id": "DEPT-B",
      "dept_name": "製造2課",
      "folder_name": "製造2課",
      "allowed_samples": {
        "RPT-001": ["SAMPLE-002", "SAMPLE-004"],
        "RPT-002": ["SAMPLE-010"]
      }
    },
    {
      "dept_id": "DEPT-C",
      "dept_name": "品質管理課",
      "folder_name": "品質管理課",
      "allowed_samples": {
        "RPT-001": ["SAMPLE-001", "SAMPLE-002", "SAMPLE-003", "SAMPLE-004", "SAMPLE-005"],
        "RPT-002": ["SAMPLE-010", "SAMPLE-011", "SAMPLE-012"]
      }
    }
  ]
}
```

**ポイント:**
- `report_definitions` で報告書名と検索条件を管理 → 設定画面で追加・編集可能
- `departments[].allowed_samples` で報告書ごと×部署ごとのサンプル閲覧権限を定義
- サンプルIDで管理するため、CSVのsample_idカラムと一致させる必要あり

### 2.3 報告条件ファイル（report_conditions.json）

各部署フォルダに格納。Excel Onlineのテンプレートがこれを参照。

```json
{
  "report_id": "RPT-001",
  "report_name": "定期環境分析報告",
  "job_number": "JOB-2026-0042",
  "protocol_name": "水質分析プロトコルA",
  "created_at": "2026-03-11T10:30:00",
  "created_by": "担当者名",
  "dept_id": "DEPT-A",
  "dept_name": "製造1課",
  "sample_count": 3
}
```

### 2.4 送信通知ファイル（notification.json）

送信ボタン押下時に各部署フォルダに格納。Power Automateのトリガー。

```json
{
  "report_id": "RPT-001",
  "report_name": "定期環境分析報告",
  "job_number": "JOB-2026-0042",
  "sent_at": "2026-03-11T14:00:00",
  "sent_by": "担当者名",
  "dept_id": "DEPT-A",
  "dept_name": "製造1課",
  "message": "3月度の水質分析結果をご確認ください。",
  "report_link": "https://会社名.sharepoint.com/sites/.../report_template.xlsx"
}
```

## 4. 処理フロー詳細

### 4.1 起動〜データ取得

```
ツール起動
  │
  ├── config.json を読み込み
  │   └── 存在しなければ初回設定画面を表示
  │
  ├── SharePoint同期フォルダからCSVを読み込み
  │   パス: config.sharepoint_paths.source_dir / analysis_data.csv
  │
  └── メイン画面を表示
      └── 報告書プルダウンに report_definitions を表示
```

### 4.2 報告書選択〜データ絞り込み

```
報告書を選択
  │
  ├── 選択した report_definition の search_filters を取得
  │
  ├── CSVデータを protocol_name でフィルタ
  │
  ├── フィルタ結果からユニークな job_number を抽出
  │   └── JOB番号プルダウンに表示
  │
  ├── JOB番号を選択
  │   └── job_number + protocol_name でCSVを絞り込み
  │
  └── ② データプレビューに表示
      └── ③ 部署別配信プレビューを自動計算
          各部署の allowed_samples と照合して件数を表示
```

### 4.3 CSV出力・プレビュー

```
[CSV出力・プレビュー] クリック
  │
  ├── 各部署について:
  │   │
  │   ├── allowed_samples に基づきCSVを分割
  │   │
  │   ├── 部署フォルダに report_data.csv を書き出し
  │   │   パス: reports_dir / {部署フォルダ名} / report_data.csv
  │   │
  │   └── report_conditions.json を書き出し
  │
  └── 「SharePointに出力しました。Excel Onlineで確認してください。」
      └── 各部署フォルダのExcelテンプレートへのリンクを表示
```

### 4.4 送信

```
[送信] クリック
  │
  ├── 確認ダイアログ
  │   「以下の部署に報告を送信します。よろしいですか？」
  │
  ├── チェックされた各部署について:
  │   │
  │   └── notification.json を部署フォルダに書き出し
  │       └── Power Automate がトリガーされる
  │
  └── 「送信完了」メッセージ表示
```

## 5. Power Automate フロー設計

```
トリガー: SharePoint — ファイルが作成されたとき
  フォルダ: /reports/{各部署フォルダ}
  フィルター: notification.json

アクション:
  1. JSONファイルの内容を取得
  2. JSONを解析
  3. フォルダの閲覧権限者リストを取得（SharePointグループ or 直接指定）
  4. メール送信（各閲覧権限者に）
     件名: 「{report_name} が送信されました — {job_number}」
     本文:
       {sent_by} より報告が送信されました。
       メッセージ: {message}
       レポートを確認: {report_link}
```

**注意:**
- notification.json は送信のたびに上書きされるため、Power Automate側で「ファイルが変更されたとき」トリガーも検討
- もしくは notification_{timestamp}.json のようにファイル名をユニークにする

## 6. Excel Online テンプレート設計

report_template.xlsx は各部署フォルダに事前配置。Power QueryでCSVとJSONを読み取る。

### 読み取り対象
- `report_data.csv` — 分析データ（同一フォルダ内の相対パス）
- `report_conditions.json` — 報告条件

### シート構成（案）
1. **報告書** — メインの表示シート（条件情報 + データテーブル）
2. **データ** — Power Queryで取り込んだ生データ（非表示推奨）
3. **条件** — Power Queryで取り込んだ条件情報（非表示推奨）

### Power Query設定
- データソース: 相対パスでCSV/JSONを参照
- 更新: ブック起動時に自動更新、または手動更新ボタン

## 9. 想定リスク・注意事項

| リスク | 対策 |
|---|---|
| SharePoint同期の遅延 | 書き込み後にファイル存在確認のリトライ処理を入れる |
| 同時に複数人がツールを操作 | 送信前にnotification.jsonの更新日時を確認し、競合を警告 |
| config.jsonの不整合 | 起動時にバリデーション、不正値は警告表示 |
| Power Automateトリガーの重複実行 | notification_*.json のファイル名にタイムスタンプを含め、処理済みフラグを持たせる |
| 部署フォルダ権限の設定漏れ | ツールに部署フォルダの存在チェック機能を持たせる |
| CSVの文字コード問題 | UTF-8 BOM付きで出力（Excel互換性のため） |

## 11. リファレンス: Bunseki_ccc

**リポジトリ:** https://github.com/h-yamamoto-8g/Bunseki_ccc

既存の `bunseki.exe` プロジェクトから以下の設計パターンを踏襲する。

### 11.1 依存構造: core → service → UI ラッパー → UI

Bunseki_ccc は以下の4層アーキテクチャを採用している。報告ツールもこれに倣う。

```
app/
├── core/           ← データアクセス層（Store / Loader）
│   ├── loader.py           CSV読み込み・クエリ
│   ├── job_store.py        JOBデータ管理
│   ├── user_store.py       ユーザーデータ管理
│   └── ...
│
├── services/       ← ビジネスロジック層（core のみに依存）
│   ├── data_service.py     DataLoader をラップしてクエリ提供
│   ├── data_update_service.py  外部ツール実行・データ更新
│   ├── job_service.py      JOB操作のビジネスルール
│   └── ...
│
├── ui/             ← プレゼンテーション層
│   ├── generated/          Qt Designer 生成コード（自動生成、手動編集しない）
│   ├── pages/              各ページの実装（wrapper パターン）
│   │   ├── home/
│   │   │   └── wrapper.py  ← generated UI を継承し、service を呼び出す
│   │   ├── tasks/
│   │   │   └── wrapper.py
│   │   └── settings/
│   │       └── page.py
│   ├── states/             タスクの各ステート画面
│   ├── dialogs/            ダイアログ（ログイン、ローディング等）
│   ├── widgets/            共通ウィジェット（サイドバー等）
│   └── styles/             QSS スタイル定義
│
└── config.py       ← アプリ設定・パス管理（モジュールレベル変数）
```

**依存の方向:**
- `core` → 外部依存なし（pandas, json のみ）
- `services` → `core` のみに依存（UI を知らない）
- `ui/pages` → `services` に依存（core を直接触らない）
- `ui/generated` → 自動生成コード。wrapper が継承して使う

**報告ツールへの適用:**

| Bunseki_ccc の層 | 報告ツールでの対応 |
|---|---|
| `core/loader.py` | CSV読み込み・フィルタ・分割ロジック |
| `services/data_service.py` | 報告データのクエリ・権限フィルタ |
| `services/data_update_service.py` | 最新CSVの取得処理 |
| `ui/pages/` | メイン画面・設定画面 |
| `config.py` | パス管理・設定ファイル読み込み |

### 11.2 データ更新処理（data_update_service.py）

起動時（ログイン後）に `run_all()` を実行してデータを最新化するパターン。報告ツールでも同様の流れを採用する。

**Bunseki_ccc のフロー:**
```
run_all()
  ├── run_data_update()     外部 Extractor exe を subprocess で実行
  │   ├── パス存在確認
  │   ├── result.json の mtime 記録（誤判定防止）
  │   ├── subprocess.run() で実行（CREATE_NO_WINDOW フラグ）
  │   └── result.json の ok/csv_path を検証
  │
  └── run_normalization()   外部 ETL exe を subprocess で実行
      ├── subprocess.run() で実行
      └── returncode で成否判定
```

**報告ツールでの適用:**
- Bunseki_ccc と同じ CSV（bunseki.csv）を使う場合、`data_update_service.run_all()` と同等の処理を起動時に呼び出す
- 別スレッド（QThread / threading）で実行し、UIをブロックしない（`_DataUpdateWorker` パターン）
- `ENABLED` フラグで一時無効化できる設計も踏襲

```python
# 報告ツールでの実装イメージ
class ReportDataUpdateService:
    """報告ツール用データ更新サービス。"""
    ENABLED: bool = True

    def run_all(self) -> None:
        if not self.ENABLED:
            return

        csv_path = config.SOURCE_CSV_PATH
        if not csv_path.exists():
            raise DataUpdateError(f"元データCSVが見つかりません: {csv_path}")
```

### 11.3 EXE化の方法

**PyInstaller spec ファイル（bunseki.spec）の構成:**

```
bunseki.spec
├── Analysis       ソース解析・依存関係収集
│   ├── datas      リソースファイルの同梱（splash.png 等）
│   ├── hiddenimports   動的インポートの明示指定
│   └── excludes   不要モジュールの除外（サイズ削減）
│
├── Splash         PyInstaller Splash（Tcl/Tk ベース、Python起動前に即表示）
│
└── EXE            単一ファイル exe 生成
    ├── console=False    GUIアプリのためコンソール非表示
    ├── upx=True         圧縮有効
    └── icon=...         アプリアイコン
```

**報告ツールでの適用:**
- PySide6 を使う場合は `hiddenimports` に QtSvg 等を追加
- tkinter/customtkinter を使う場合はよりシンプルな spec で済む
- `--onefile` でも `--onedir` でも可（OneDrive同期の問題がある場合は `--onedir` + launcher.bat）

### 11.4 起動前のスプラッシュ画面

Bunseki_ccc は **2段階スプラッシュ** を実装している:

```
exe起動
  │
  ├── [1] PyInstaller Splash（Tcl/Tk ベース）
  │   └── Python インタプリタが起動する前に即座に表示
  │       bunseki.spec の Splash() で設定
  │
  ├── Python 起動・重いモジュールの遅延インポート
  │   └── _load_app_modules() でスプラッシュ表示中にインポート
  │       qapp.processEvents() で UI を止めない
  │
  └── [2] QSplashScreen（開発環境用）
      └── frozen 環境でなければ Qt の QSplashScreen を表示
          _SPLASH_MIN_MS = 1500ms の最低表示時間を確保
```

**報告ツールでの適用:**
- spec ファイルに `Splash()` を追加するだけで PyInstaller Splash を実現
- 重いインポート（pandas 等）はスプラッシュ中に処理
- tkinter 採用の場合は PyInstaller Splash のみで十分（QSplashScreen 不要）

### 11.5 データパスの管理方法

Bunseki_ccc は `config.py` でパスを一元管理する:

```
パス解決の優先順位:
  1. ~/.bunseki/settings.json の app_data_path（ユーザーが設定画面で指定）
  2. SharePoint 同期フォルダの標準パス（Windows: ~/トクヤマグループ/環境分析課 - ドキュメント/app_data）

DATA_PATH（ルート）
  ├── _common/
  │   ├── master_data/source/    マスタデータ
  │   ├── data/lab_aid/normalized/bunseki.csv   正規化済みデータ
  │   └── tools/                 外部ツール配置
  ├── bunseki/
  │   ├── tasks/                 タスクデータ
  │   └── logs/                  ログ
  └── ...
```

**報告ツールでの適用:**

```python
# config.py（報告ツール版）
LOCAL_SETTINGS_DIR = Path.home() / ".houkoku"
LOCAL_SETTINGS_PATH = LOCAL_SETTINGS_DIR / "settings.json"

def _resolve_data_path() -> Path:
    """settings.json → SharePoint標準パスの順で解決。"""
    saved = load_data_path()
    if saved is not None:
        return saved
    if platform.system() == "Windows":
        return Path.home() / "トクヤマグループ" / "環境分析課 - ドキュメント" / "報告ツール"
    return Path.home() / "houkoku_data"

DATA_PATH = _resolve_data_path()

# 派生パス
SOURCE_CSV_PATH = DATA_PATH / "source" / "analysis_data.csv"
REPORTS_PATH = DATA_PATH / "reports"
CONFIG_PATH = DATA_PATH / "config"
```

**ポイント:**
- `reload_paths(new_path)` 関数でモジュールレベル変数を動的に更新できる設計
- 初回起動時にパスが見つからなければ `SetupRootDialog` 相当のダイアログで設定させる
- Bunseki_ccc と同じ SharePoint チームサイトの同期フォルダ配下に配置する想定
